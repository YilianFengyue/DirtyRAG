## 1. 一句话：这个项目在干嘛

我们做一个**对比实验台**：

> 同一批问题、同一批文档、同一个大模型，让 5 种"答题策略"各答一遍，
> 看谁在**脏数据（文档互相矛盾、有错误信息、有噪声）**下答得更稳。

我们自己的方法叫 **EvidenceBoard-RAG**，它不直接把文档塞给大模型，而是先把文档加工成"证据卡"、建一张"冲突关系图"，再决定怎么答（或者要不要弃答）。

---

## 2. 为什么要做（背景）

普通 RAG（检索增强生成）的流程是：

```
问题 → 检索一堆文档 → 全部拼起来塞给大模型 → 出答案
```

它只关心"文档跟问题相不相关"，**不管文档之间是不是互相打架**。

但真实知识库很脏：
- 同一个问题，不同文档给出**互相矛盾**的答案
- 有些文档是**错误信息**（misinfo）
- 有些是**无关噪声**（noise）
- 同一个错误答案被**重复**好几遍，模型就以为它更可信
- 同名实体混淆（比如两个叫 Michael Jordan 的人）

普通 RAG 在这种场景下很容易被带偏。我们要解决的就是这个"**检索之后、生成之前**"的鲁棒性问题。

---

## 3. 五个"答题选手"（方法）

| 选手 | 一句话策略 | 角色 |
|---|---|---|
| **Direct LLM** | 啥文档都不看，纯靠模型记忆答 | 下限对照 |
| **Vanilla RAG** | 把全部文档一股脑塞给模型 | 最基础 RAG |
| **RAG + Filter** | 先让模型判每篇文档有没有用，没用的扔掉再答 | 相关性过滤 |
| **CRAG-style** | 先评估"这批文档整体质量如何"，差就过滤/弃答 | 检索纠错（强 baseline） |
| **EvidenceBoard-RAG（我们的）** | 抽证据卡 → 找重复 → 建冲突图 → 防稀释打分 → 验证器复核 | 主角 |

前 4 个是对照组，第 5 个是我们提出的方法。

---

## 4. 我们的方法 EvidenceBoard-RAG 怎么工作

5 步流水线：

```
1. 抽证据卡：每篇文档单独问模型"你支持什么答案？相关度多高？"，得到结构化卡片
2. 找重复  ：文本高度相似的文档归为一组，避免重复证据靠"刷量"取胜
3. 建冲突图：判断文档两两关系——支持 / 矛盾 / 重复
4. 防稀释打分：按"答案"聚合而不是按"文档数"投票，重复的只算一次
5. 验证器  ：候选答案交给模型再复核一遍，决定 答案 / 修正 / 冲突 / 不知道
```

最后每道题都会存一份"证据板 JSON"，完整记录它是怎么从一堆乱文档推出最终答案的——**这就是论文 case study 的素材**。

## 5. 目前最好的结果（RAMDocs 200 条）

完整数据见 `docs/DirtyRAG_实验报告_RAMDocs200.md`。核心：

| 指标 | 最强 baseline | 我们的方法 | 说明 |
|---|---|---|---|
| 严格成功率 strict_success | 0.36 | **0.495** | 主指标，相对提升约 37% |
| 正确答案覆盖 gold_coverage | 0.40 | **0.505** | 答得更全 |
| 冲突识别 conflict_sensitivity | 0.195 | **0.35** | 接近翻倍 |
| 错答泄漏 wrong_leakage | 0.01（CRAG） | 0.06 | 这项没赢，但失败以"保守弃答"为主 |
| token 成本 | 1091（Vanilla） | 3596 | 代价是约 3.3 倍 |

**三句话总结结果：**
1. 我们的方法在 strict success / gold coverage / conflict sensitivity 上明显领先。
2. wrong_leakage 没赢，但拆开看：我们的失败 73% 是主动说"冲突/不知道"，只有 12% 是真错答——**脏知识库下"宁可不答不可误导"更安全**。
3. 代价是约 3.3 倍 token、4.5 倍延迟。**我们不声称全面 SOTA**。

还有个有意思的发现：**CRAG-style 反而比 Vanilla 还差**（0.22 vs 0.335），因为它把"文档有冲突"误判成"质量不足"而过度弃答——这恰好证明了"相关性纠错不够，需要冲突建模"，是我们论文的核心动机。
