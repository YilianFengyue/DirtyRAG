## 2. 主结果表（直接抄进论文）

`paper_results/ramdocs_200_final/metrics.csv` 内容：

| 方法 | strict_success | gold_coverage | wrong_leakage | conflict_sensitivity | avg_latency(s) | avg_total_tokens |
|---|---:|---:|---:|---:|---:|---:|
| Direct LLM | 0.050 | 0.055 | 0.010 | 0.000 | 2.66 | 141 |
| Vanilla RAG | 0.335 | 0.380 | 0.060 | 0.180 | 7.26 | 1091 |
| RAG + Filter | 0.360 | 0.400 | 0.055 | 0.195 | 19.10 | 2528 |
| CRAG-style | 0.220 | 0.220 | 0.015 | 0.140 | 9.45 | 2025 |
| **EB-RAG (Ours)** | **0.495** | **0.505** | 0.060 | **0.350** | 33.06 | 3596 |

指标含义：
- **strict_success**（主指标，越高越好）：答案覆盖 gold **且**没泄漏 wrong
- **gold_coverage**（越高越好）：答案命中任一 gold answer（宽松匹配）
- **wrong_leakage**（越低越好）：答案里出现了错误答案
- **conflict_sensitivity**（越高越好）：答案显式标识了冲突/歧义
- **avg_latency / avg_total_tokens**：每题平均耗时 / token（成本）

> ⚠️ gold_coverage 是"命中任一 gold 即算"的宽松匹配，多答案题会偏乐观，论文里写一句说明即可。

## 4. 每个指标数字是怎么算出来的（给写实验方法学的）

评分逻辑在 `src/dirtyrag/evaluation/metrics.py`，核心函数 `score_prediction`：

1. 取模型回答 `answer`、该题的 `gold_answers`、`wrong_answers`。
2. `has_gold` = 回答里出现任一 gold answer（归一化后子串匹配）。
3. `has_wrong` = 回答里出现任一 wrong answer。
4. `conflict_flag` = 回答里出现冲突关键词（conflict / contradict / inconsistent / multiple answers / cannot determine）。
5. 由此得：
   - strict_success = has_gold AND (NOT has_wrong)
   - gold_coverage = has_gold
   - wrong_leakage = has_wrong
   - conflict_sensitivity = conflict_flag
6. 方法级数字 = 该方法 200 题的平均（`summarize_metrics`）。

> 这是**字符串匹配**评分，没用 LLM judge（`use_llm_judge: false`），优点是确定、可复现，缺点是对措辞敏感、宽松匹配。论文写 evaluation 时如实说明即可。

## 5. EvidenceBoard-RAG 内部行为（写分析/case study 用）

逐条统计（200 条，来自实验报告 §4）：

```
逐条胜负：
  所有方法都答对        : 1 条
  EB-RAG 独有答对       : 43 条   ← 4 个 baseline 全错、只有我们对
  所有方法都答错        : 72 条   ← RAMDocs 极难，绝对天花板低

EB-RAG 回答分类（200 条）：
  答对 strict success      : 99 条 (49.5%)
  保守弃答 conflict/unknown : 73 条 (36.5%)  ← 失败主因是"不答"而非"答错"
  其他未命中                : 16 条 (8%)
  错答泄漏 wrong leak       : 12 条 (6%)

决策模式（200 个证据板）：
  candidate: answer 128 / conflict 70 / unknown 2
  verifier : supported 127 / conflict 70 / unknown 3
```

**怎么查单条 case 的内部过程**：打开 `paper_results/ramdocs_200_final/evidence_boards/<qid>_evidenceboard_rag.json`，里面有：
- `cards`：每篇文档抽出的证据卡
- `duplicate_groups`：重复分组
- `conflict_edges`：冲突图的边（support/contradict/duplicate）
- `answer_clusters`：答案簇 + 打分
- `candidate_decision` / `verifier_decision`：两阶段决策
- `decision_trace`：人类可读的决策日志
- `final_answer`：最终答案

## 7. 怎么自己重跑（如果需要验证或加实验）

### 7.1 装环境
```bash
python -m venv .venv
.\.venv\Scripts\Activate.ps1        # Windows
# 或 source .venv/bin/activate       # Linux
pip install -r requirements.txt
pip install -e .
pytest -q                            # 应 13 passed
```

> Linux 服务器若 pip 编译 pyarrow/libcst 失败，用 conda：
> `conda create -n dirtyrag -c conda-forge python=3.10 pyarrow datasets openai pydantic python-dotenv pyyaml tqdm pytest -y`
> 再 `pip install -e . --no-deps`

### 7.2 配 API（DeepSeek）
```bash
# .env 文件（不要提交 git）
LLM_API_KEY=你的key
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_MODEL=deepseek-chat
```

### 7.3 准备数据
```bash
python -m dirtyrag.data.prepare_datasets --dataset ramdocs --limit 200
# 生成 data/processed/ramdocs_200.jsonl
# （本地若有 datasets/RAMDocs/raw/ramdocs/repo/RAMDocs_test.jsonl 会优先读本地，不联网）
```

### 7.4 跑实验
```bash
# 先冒烟 1 条验证链路
python -m dirtyrag.cli run --config configs/ramdocs_200.yaml --limit 1

# 全量 200（建议 nohup 后台）
nohup python -m dirtyrag.cli run --config configs/ramdocs_200.yaml --limit 200 > run.log 2>&1 &
```

### 7.5 评估 + 画图
```bash
python -m dirtyrag.cli evaluate --run-dir outputs/runs/latest
python -m dirtyrag.cli plot --run-dir outputs/runs/latest
# 图出在 outputs/runs/latest/figures/
```

> 跑中途崩了不用从头：`--resume-run outputs/runs/run_xxx` 会跳过已完成的 (题, 方法)。

## 9. 给写论文的两个提醒

1. **不要写的话**（会被老师/师兄抓）：
   - ❌ "zero wrong-answer leakage"（10 条时的假象，200 条是 0.06）
   - ❌ "outperforms all baselines on all metrics"（wrong_leakage 输给 CRAG/Direct）
   - ❌ "fully reproduces CRAG"（我们是简化的 CRAG-style，去掉了 web search 和训练版 evaluator）

2. **三个最强卖点**（实验报告 §6 有英文写法）：
   - strict success 相对提升 ~37%，43 条独有答对
   - conflict sensitivity 接近翻倍
   - 失败以保守弃答为主（73 弃答 vs 12 错答），脏库下更安全
   - 附加 motivation 证据：CRAG-style 反而比 Vanilla 差（相关性纠错不足以处理冲突）
